# 脱稿思维汇报法

**R（Reading）— 原文引用**
> 用脱稿思维为他起草汇报材料，才能写成符合领导站位和逻辑的样子

**I（Interpretation）— 理解重写**
脱稿思维是假设领导不带稿子，用脱稿思维起草汇报材料的思维方式。

汇报材料不是拿来发表传阅存档的文献，而是服务于人和人面对面、在规定时间内完成一次有效的信息交换和价值传递。

脱稿思维六维度：
1. 开头：直截了当点题切入（一句话结构化表达锁定汇报重心）
2. 结构：领导抓工作逻辑（上级最不放心的风险点先兜底/硬任务拿数据说话/历史遗留问题讲突破性进展）
3. 标题：实实在在工作维度切分（不用对仗工整的书面语和过多修辞）
4. 内容：不堆砌苦劳写成次数（写机制突破点/写点上落实）
5. 数据：三维筛选（一针见血的数据）：上级最关注的数据/本单位亮点数据/系统外部反馈良好的数据
6. 问题：不写泛化表述（写卡在具体环节上的真问题，需要领导协调解决的事项）

**A1（past case）**
有稿思维 vs 脱稿思维：
- 有稿（开头）："近年来在某某的正确领导下，在某某的大力支持下..."
- 脱稿（开头）："今天重点汇报三件事：一是...二是...三是..."
- 有稿（结构）：摸清底数/制定方案/分步实施/总结验收
- 脱稿（结构）：上级最不放心的风险点先兜底/硬任务拿数据说话/历史遗留问题讲突破性进展

**A2（future trigger）**
你要为领导写一份在市里经济工作会上的汇报，需要用脱稿思维。

**E（Execution）— 执行步骤**
1. **开头**：直截了当点题，不要背景铺陈
2. **结构**：按领导抓工作逻辑（风险点/硬任务/历史遗留）组织
3. **标题**：用工作维度切分，不用对仗修辞
4. **内容**：写机制突破和点上落实，不写苦劳次数
5. **数据**：三维筛选（上级关注/本单位亮点/外部反馈）
6. **问题**：写卡在真问题上的具体事项，不写泛化表述

**B（Boundary）— 边界与局限**
- **失效场景**：
  - 不是面对面汇报的场合（书面报告、正式文件不适合脱稿思维）
  - 领导本人已确定讲话风格的正式场合（需要遵循既定风格）
  - 需要留存存档的正式文献
- **常见错误**：
  - 把"脱稿"当成"随便"——脱稿思维是更精准的适配，不是粗糙随便
  - 开头仍然使用背景铺陈（"近年来在XX领导下..."）
  - 结构按部门做事逻辑而非领导抓工作逻辑
  - 堆砌苦劳和次数（"开展了XX次会议"、"完成了XX次检查"）
  - 数据不加筛选（罗列所有数据而非一针见血的数据）
  - 问题泛化（"建议加强领导"而非"卡在XX环节的XX问题"）
- **作者盲点**：
  - 脱稿思维的核心是"场景适配"，不是"删繁就简"
  - 脱稿思维的检验标准是：领导能不能不带稿子直接念
  - 脱稿思维写的不是"有没有"，而是"能不能说"
